# BRMSTE Networks — `brmste.com/networks`

The production page for **BRMSTE Networks**. It is a single self-contained
`index.html` that pulls **live** Bitcoin & Lightning telemetry **client-side**
from [mempool.space](https://mempool.space) (CORS-enabled), so it needs **no
backend** and works the moment it is served on `brmste.com/networks`.

- Vision constant `8^8 = 2^24 = 16,777,216` is self-verified in the browser.
- Live Lightning node/channel/capacity figures + the declared on-chain address
  status (shown honestly — an empty address reads "Unfunded").
- Brand-matched to the BRMSTE gold / emerald / obsidian system.

## Files

| File | Purpose |
|------|---------|
| `index.html` | The page (self-contained: inline CSS/SVG/JS, no dependencies) — **single source of truth** |
| `canonical-route.js` | Zero-binding drop-in `/networks` route for the **main** worker (auto-generated from `index.html`) |
| `worker.js` | Standalone Cloudflare Worker that serves the page at `/networks` (uses an `ASSETS` binding) |
| `wrangler.toml` | Standalone-worker config + route `brmste.com/networks*` |

`canonical-route.js` is generated by [`scripts/build-networks-route.py`](../../scripts/build-networks-route.py)
so it can never drift from the page. Re-run after editing `index.html`; CI enforces
sync with `python3 scripts/build-networks-route.py --check`.

## Live edge map (verified via the Cloudflare MCP server)

`brmste.com` is fronted by Cloudflare. Mapping the account with the Cloudflare MCP
tools (`workers_list`, `workers_get_worker`) shows:

| Fact | Value |
|------|-------|
| Workers in account | **19** |
| Canonical site worker | `brmste-canonical-leader-router` (id `905109fa4cb04c15a0d66c4e66479a66`) |
| `brmste-networks` worker | **not present yet** — so `/networks` is unrouted and currently **404s** |
| Page-content stores | none wired for the site (no Workers-Sites KV / D1; R2 not used for pages) — the canonical worker emits **HTML inline** |

Because the canonical worker builds its HTML in code (not from an asset/KV/D1/R2
store), the most faithful "main business" integration is the inline drop-in route
in `canonical-route.js` — **Option A** below.

> Note: the Cloudflare MCP server is **read-only for Workers** (it can read
> `workers_list` / `workers_get_worker` / `workers_get_worker_code`, and write only
> D1 / KV-namespace / R2-bucket *metadata*). It cannot deploy or edit Worker code,
> so going live still needs a Cloudflare API token or the site repo — see below.

## Deploy — pick one

### Option A — patch the canonical worker (this is our main business)

Add `/networks` directly to `brmste-canonical-leader-router` — the worker that
already serves `brmste.com`. It emits HTML inline, so the drop-in route in
[`canonical-route.js`](./canonical-route.js) needs **no binding** (no ASSETS / KV
/ D1 / R2). Add `canonical-route.js` to the worker's source, then two lines at the
very top of its `fetch` handler, *before* the SPA/catch-all route:

```js
import { handleNetworks } from "./canonical-route.js";   // or paste the file inline
// ...
async fetch(request, env, ctx) {
  const networks = handleNetworks(request);
  if (networks) return networks;          // serves /networks; null for everything else
  // ...existing canonical routing continues unchanged...
}
```

`handleNetworks` returns a `Response` only for `/networks` and `/networks/`, and
`null` for every other path — so the rest of the site is untouched. Verified
locally: `/networks` → `200 text/html` (full page), all other paths → pass-through.
Deploy the canonical worker the way that repo already deploys (the source is at
`gitlab.com/brmste/brmste`, private).

### Option B — standalone Cloudflare Worker route (no SPA changes)

Because `brmste.com` is already fronted by Cloudflare, a Worker bound to the
route `brmste.com/networks*` intercepts that path at the edge **before** the SPA
origin — so you add `/networks` without modifying the main site repo.

```bash
cd site/networks
npx wrangler login            # or set CLOUDFLARE_API_TOKEN + CLOUDFLARE_ACCOUNT_ID
npx wrangler deploy
```

Or run the **Deploy BRMSTE Networks (Cloudflare)** GitHub Action
(`.github/workflows/deploy-networks.yml`) after adding these secrets:

- `CLOUDFLARE_API_TOKEN` — token with *Workers Scripts: Edit* and access to the
  `brmste.com` zone (*Workers Routes: Edit*).
- `CLOUDFLARE_ACCOUNT_ID` — the account that owns the `brmste.com` zone.

This publishes a new `brmste-networks` worker (currently absent — see the edge map
above) bound to `brmste.com/networks*`.

### Option C — drop into the brmste.com site repo (Vite/SPA)

The brmste.com source lives at `gitlab.com/brmste/brmste` (private) and is a
**Vite + React** app. Vite copies everything in `public/` verbatim to the build
root, so:

```bash
# inside a checkout of gitlab.com/brmste/brmste
mkdir -p public/networks
cp /path/to/.github/site/networks/index.html public/networks/index.html
git add public/networks/index.html && git commit -m "feat: add /networks page" && git push
```

The page is fully standalone (no imports), so it needs no build wiring.

**SPA caveat:** if the host serves `index.html` for *all* unknown routes
(SPA fallback), make sure a real file at `/networks/index.html` takes
precedence. On Cloudflare Pages / Netlify / Vercel a concrete file wins over the
fallback automatically. If your host 200s the SPA for `/networks`, use
**Option A or B** (which intercept before the SPA) instead.

### Option D — Cloudflare Pages project + route

Deploy this folder as a Pages project and map `brmste.com/networks` to it via a
Pages custom-domain path / Cloudflare route.


Then open `https://brmste.com/networks` — the tiles populate from
`mempool.space` within a second and refresh every 60s.
